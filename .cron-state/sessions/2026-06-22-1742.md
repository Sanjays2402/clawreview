# Tick 21 — 2026-06-22 17:24 PT

5 features shipped end-to-end. Theme: round out tick 20's `?minConfidence`/
`?severityThreshold` filter knobs with the full observability + UX stack.

## What landed (in commit order)

1. **`706a742` — aggregator: `findingDigestWithFilterReport`**
   - Wrapper around findingDigest() that surfaces inputTotal /
     droppedTotal / appliedFilters (raw + normalised + applied bit per
     filter, plus `any`). Digest body byte-identical (pinned).
   - Three consumers below all depend on this single source of truth
     for the "did the filter apply, how much did it drop" question.
   - 10 new tests in digest.test.ts.

2. **`1145ee0` — telemetry: `clawreview_review_digest_filter_applied_total`**
   - Closed yes|no labels on both `min_confidence` and
     `severity_threshold`, cardinality cap of 4. Cheap counter that
     answers "what fraction of /digest reads filter?".
   - New `REVIEW_DIGEST_FILTER_APPLIED_LABELS` tuple +
     `deriveReviewDigestFilterAppliedLabel(applied)` predicate +
     `observeReviewDigestFilterApplied(metrics, minApplied, sevApplied)`
     all exported.
   - 8 new tests.

3. **`91a2560` — server: /api/reviews/:id/digest fires the counter**
   - Swaps the fresh-arm digest call from findingDigest() to
     findingDigestWithFilterReport() (zero perf cost; pure wrapper).
   - Fires the new counter on the way out alongside the existing
     drift counter, reusing the metrics bundle.
   - Cached arm intentionally inert -- persisted digest carries no
     filter metadata, so counting cached reads would corrupt the
     ratio.
   - 7 new tests (counter group).

4. **`9eb332a` — server: ?normalisedEcho=true clamped-value echo**
   - Opt-in echo of `normalisedMinConfidence` + `normalisedSeverityThreshold`
     alongside the raw operator-supplied values (default OFF for
     back-compat). Works on both cached + fresh arms.
   - Headline: `?minConfidence=1.5&normalisedEcho=true` -> raw=1.5 +
     normalised=1, so dashboards render "confidence >= 1 (clamped
     from 1.5)" without re-running the digest's normaliser.
   - New `parseNormalisedEchoFlag(raw)` pure helper exported.
   - 12 new tests (8 route integration + 4 parser unit).

5. **`b01309e` — cli: stats --filter-summary text-mode header**
   - Opt-in one-line text header: "Showing N findings (filtered M of
     K by min_confidence >= 0.5 + severity_threshold >= medium)".
     Uses NORMALISED values (matches the server's ?normalisedEcho
     surface so the two agree).
   - Default OFF (text shape unchanged). JSON output unaffected --
     tick-20 echo already serves machine consumers.
   - Internal swap: stats now uses findingDigestWithFilterReport so
     the drop count is free.
   - New `renderFilterSummaryLine(report, c)` pure helper exported.
   - 8 new tests.

## Notes

- The five features composed naturally: helper -> counter -> server
  wiring -> server echo opt -> CLI text header. Each commit was
  small (~200-260 lines), revertible, and tested. The pattern of
  building the helper first then layering consumers on top kept
  each commit's diff focused.
- Used `findingDigestWithFilterReport`'s `appliedFilters` bits as
  the single source of truth for "did the filter apply" across
  telemetry + server echo + CLI text header. No drift between the
  three surfaces.
- The metrics-test discovery (server's /metrics endpoint closure-
  captures the bundle at boot, so tests need to read
  `getMetrics().registry.metrics()` directly after reset rather
  than `app.inject('/metrics')`) was a 30-second debug that saved
  rewriting 6 tests later -- the existing tick-13 drift-counter
  test already used the same pattern, copying it kept the test
  surface uniform.
