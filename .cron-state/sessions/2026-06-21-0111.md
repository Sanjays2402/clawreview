## Story of the tick

**1. `findingDigest`** is the foundational primitive: the CLI's stats command,
the worker's PR-comment header, and the dashboard feed all maintained their own
inlined for-loops over Finding[] to compute totalsBySeverity / byCategory /
byAgent / byFile / topFiles. Three implementations slowly drifting apart is a
quiet correctness bug waiting to bite. Pulled it up to @clawreview/aggregator
as a single-pass helper with a fixed-shape FindingDigest result, an opt-in
hotspots field (omitted when not requested so JSON consumers can distinguish
"didn't compute" from "computed and found none"), and a clamped topFiles cap.

**2. `stats --by file` + `--top-files <n>`** lands directly on top of #1: the
CLI now consumes findingDigest instead of inlining its own loops, the JSON
shape gains `byFile`, the text output gains a "By file [(top N of M)]:" primary
block when --by file is selected (with the secondary "Top files:" footer
suppressed so the same numbers don't print twice), and --top-files bounds both
the text block and the JSON array with a clamp into [1, 200].

**3. `presets resolve <chain>`** completes the presets sub-command trinity:
`list` (what exists?), `show <name>` (what does THIS preset look like?),
`resolve <chain>` (what does THIS COMBINATION of presets compose into?). Accepts
both positional and --chain forms, surfaces unknown names / cycles / empty
entries with the full chain in the error message, and renders yaml / json /
text. Sources block attributes each chain entry as local / builtin / unknown
with shadowsBuiltin flagged so an operator can audit which preset wins on a
name collision.

**4. peak bucket** adds `peakBucketIndex` + `peakBucketCount` to the webhook
stats `hourly` block so a dashboard sparkline tooltip can label the peak
without re-walking the array client-side. Tie-break biases toward the NEWER
bucket (smaller index) so the "peaked at" label flags the most recent surge —
that's the one on-calls actually want to investigate. Null + 0 when every
bucket is empty so a consumer can fixed-render the label without conditionals.

**5. operator-poll `?force=1` bypass** is the quality-of-life knob: a
dashboard's in-band health probe (every 1-2s) was eating the operator-poll
bucket and 429ing the genuine UI polling. `?force=1` is the explicit opt-out.
Bypass is class-only — the default per-token limiter still observes the hit,
so a misbehaving client cannot hide behind force=1 forever. New
`x-ratelimit-operator-bypass: force` response header makes the bypass auditable
in dev tools / server logs without parsing the body.
