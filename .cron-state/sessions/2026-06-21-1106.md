## What landed

1. **c0d5bce — server: `/api/internal/webhook/stats?bucketWindow=` end-time anchor override.**
   Postmortem-snapshot mode for the sparkline. The store already
   accepted `nowMs` for deterministic tests; this surfaces it through
   the route as `?bucketWindow=<unix-ms>` (cheap from code) and
   `?bucketWindowAt=<ISO-8601>` (friendly from a browser URL bar).
   Deliveries with `receivedAt > anchor` are skipped from the
   sparkline (negative age) but still count toward the absolute
   `total` so the store-size banner stays honest. NaN / negative
   anchor values fall back to the live clock silently rather than
   rendering an empty sparkline by accident. `appliedFilters.bucketWindow`
   echoes `null` when defaulted vs ms when overridden so a dashboard
   can show a "Live ↔ Snapshot of <time>" toggle without re-parsing
   the query string. Composes cleanly with tick-9's peak-bucket and
   tick-7's granularity / buckets knobs — a 120-bucket minute-granular
   anchor on the two hours leading up to an incident peaks on
   whichever minute actually had the burst, not chronologically-recent
   now. **5 new tests** (webhook-replay.test.ts bucketWindow group).

2. **2f826f0 — worker: findingDigest hand-off (compute once, share with renderer + store).**
   Tick 11 wired `renderPrComment` to ACCEPT a `digest` opt; the worker
   still let the renderer build its own. This closes the loop: the
   worker computes `findingDigest(aggregated.findings, {topAgents:8,
   topCategories:8})` exactly once AFTER suppression, then hands the
   SAME reference to BOTH `renderPrComment({digest})` AND
   `store.complete(..., {digest})`. `ReviewRecord` gains optional
   `digest?: FindingDigest`; `/api/reviews/:id` DTO surfaces it
   verbatim so the dashboard detail page renders byte-identical
   totalsBySeverity / byCategory / byAgent / topFiles / topAgents /
   topCategories without re-walking findings. Legacy completes (failed
   reviews, pre-tick-12 reruns) leave the field undefined and the DTO
   exposes it as `null` so a dashboard's "has counts?" check is a
   single comparison. **4 new tests** (2 store digest persistence + 2
   DTO surface).

3. **42e176b — telemetry: `clawreview_operator_poll_bypass_total{probe,reason}` attribution counter.**
   Volume metric (`operatorPollTotal{result=bypass}`) answers "how
   many bypasses?"; this new counter answers "what authorised each
   bypass?" so a security audit can graph
   `rate(clawreview_operator_poll_bypass_total[1h]) by (reason)` for
   drift in the bypass surface separately from raw volume. Reason set
   is intentionally tiny today: closed set `['force']`. A new
   authorised bypass path (internal-network shortcut, signed hash-tag,
   etc.) MUST extend the constant in a code change a security reviewer
   can spot in the PR diff — the closed-literal type prevents a typo'd
   reason from silently fragmenting the series across two values that
   both look correct in code review (force vs frce). Rate-limit hook's
   bypass arm now fires BOTH counters (volume + attribution) on the
   same request so an operator can reconcile them per-probe. **8 new
   tests** (5 telemetry pure: bypass record / (none) bucket / closed
   literal pin / volume-vs-attribution reconcile / bundle cache;
   3 wired through Fastify: force=1 bumps both / anonymous (none)
   bucket / non-bypass leaves attribution at zero series).

4. **7cd9abc — cli: `clawreview presets diff --exclude-fields <a,b,c>`.**
   Mirror of tick-11's `--only-fields`. Same parser (trimmed, de-duped,
   empty-intermediate-entry rejected), opposite set semantics (drop
   keys IN the set instead of restricting to them). For a wide preset
   rebase where the operator wants "everything EXCEPT these noisy
   fields" — usually fields known to drift for unrelated reasons. The
   two filters are mutually exclusive at the CLI layer: combining
   them would double-filter (only -> exclude) and silently produce a
   surprising result; the CLI refuses with `--only-fields and
   --exclude-fields are mutually exclusive; pick one` at exit code 2.
   `filterPresetDeltaExcluding` is a separate pure helper (mirror of
   `filterPresetDelta`) so a future tick can add a third filter shape
   without re-walking branch logic. JSON / YAML / text renderers all
   surface the active filter for downstream tooling and the distinct
   "(no differences outside the excluded fields)" text hint
   distinguishes "filter hid drift" from "chains agree everywhere".
   **11 new tests** (7 CLI integration + 4 pure helper).

5. **0143075 — cli: `clawreview presets diff --output <path>`.**
   For migration-ticket flows where the diff body lands on disk for a
   follow-up commit. Today the operator pipes stdout (`> diff.json`)
   which interacts oddly with kleur color codes if `--no-color` isn't
   set, and CI scripts have to either strip stdout noise or run with
   `--no-color` even when they explicitly asked for JSON. `--output`
   is the friendlier first-class artifact path. Path resolution:
   relative paths land under `--root` (or cwd when --root absent) via
   `resolvePresetDiffOutputPath`, so a CI checkout that pins --root
   sees a stable file path. Absolute paths bypass the resolve. mkdir
   -p on intermediate dirs so `--output reports/2026-06-21/diff.json`
   works without a separate mkdir. stderr surfaces a single "wrote N
   bytes to <path>" confirmation. `--format text` + `--output` exits
   2 up front (text is for terminal display, not artifacts; refusing
   loudly beats writing ANSI escapes to a file). --only-fields /
   --exclude-fields compose naturally; the scoped body is what lands
   on disk. **8 new tests** (6 CLI integration + 2 pure helper).
