## Design choices worth remembering

**Severity threshold semantics mirror minConfidence.** Same forgiving
normalisation policy (unknown / mis-cased -> null = no filter), same
pre-bucket application (drops findings BEFORE any totals are
computed), same AND composition with minConfidence. One pattern; two
axes; consistent everywhere.

**Cached arm IGNORES filters but echoes them.** When a /digest
request comes in with `?recompute=cached&minConfidence=0.7`, the route
serves the persisted shape unchanged (re-filtering would silently
produce a different shape than the dashboard / CLI / PR comment
header read at write time). The filters ARE echoed so an operator
who mistakenly combined them can SEE they were inert. Same pattern
the slim projection uses on the cached arm.

**Raw operator value is echoed, not the normalised value.** A typo
like `?severityThreshold=Critical` (wrong case) normalises to null
internally, but the response echoes `"severityThreshold": "Critical"`
so a CI gate can detect the typo by comparing what was sent vs what
came back. The same convention applies to the CLI's JSON output.

**Review drift carries an explicit warning for the unsupportable
case.** When the input is /digest shape (carries fresh, no findings),
client-side filter application is impossible. Rather than silently
failing or recomputing wrong, the CLI prints a clear stderr warning
that hints at the right answer: re-request the server with
?minConfidence=... The flag still echoes in the JSON output for CI
detectability.
