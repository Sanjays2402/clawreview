## Notes

The whole tick centred on closing the loop on tick-21's `findingDigestWithFilterReport` helper.
That tick built the wrapper but only the server's /digest route consumed it; tick 22 wires it
all the way through:

1. **Worker (8f0bb3f)** -- swaps bare findingDigest() for the wrapper and persists the
   appliedFilters + inputTotal + droppedTotal on ReviewRecord.filterReport so the dashboard
   can answer "was this snapshot filtered?" without recomputing.
2. **Server DTO (8f0bb3f, 11d1a50)** -- /api/reviews/:id surfaces the persisted filterReport;
   /api/reviews/:id/digest gains an opt-in ?filterDropEcho=true for the "filtered M of K"
   inline label.
3. **Telemetry (c539149)** -- worker-side counter pairs with tick-21's read-side counter
   so dashboards can chart "writes that filter" vs "reads that filter" drift.
4. **CLI (63471e5, 3c3e1b2)** -- stats gets --json-header for jq-pipeline short-circuits;
   review drift gets --base/--target for the "did the bug fix clear the findings?" loop.

The two-review compare command (3c3e1b2) is the headline UX win: 1 line of cli to answer
"did the fix work?" instead of curl + jq + manual diff. Composes with the tick-20 filter
flags so an operator can preview drift at any threshold without touching the server.

Three carried items remain blocked on dashboard wiring (apps/dashboard is Next.js, outside
the unit-test-driven loop). Refilled tick 23 with 5 fresh items including the
review drift --on-regression hook (mirror of the --watch --on-drift pattern, scoped to
compare mode) and a standalone /filter-report endpoint for dashboards that only need the
drop count.
